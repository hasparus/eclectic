# Bachanalia Fantastyczne — new website plan

Planning notes for rebuilding <https://bachanaliafantastyczne.pl/> as a Next.js
site on Vercel, with the existing WordPress kept as a headless CMS and
[ludamus](https://github.com/zagrajmy/ludamus) plugged in for the convention
programme.

This will eventually live in its own repo — these notes are just planning.

Context: Bachanalia Fantastyczne is run by ZKF Ad Astra in Zielona Góra.
The 2026 edition is the 40th ("XL"), **Sept 25–27, 2026**, and holds Polcon
rank — so there's a hard deadline and probably more traffic than usual.
The site is Polish-only, which conveniently lets us skip headless i18n
(Polylang/WPML) entirely.

Research details with sources: [research/current-site.md](research/current-site.md),
[research/ludamus.md](research/ludamus.md),
[research/wordpress-headless.md](research/wordpress-headless.md).

## Architecture

```
bachanaliafantastyczne.pl        → Vercel (Next.js App Router)
wp.bachanaliafantastyczne.pl     → existing WordPress hosting (editors only)
bachanalia.zagrajmy.net (?)      → ludamus sphere (programme + enrollment)
```

- **Next.js (App Router) on Vercel** renders everything public. Pages are
  static with tag-based ISR; nothing rebuilds on publish, WordPress pings a
  revalidate route instead.
- **WordPress stays where it is**, moved to a subdomain, frontend redirected
  to the new site. Editors keep the wp-admin they already know. The WP theme
  stops mattering entirely.
- **Ludamus owns the programme data.** The website renders a read-only
  programme grid server-side and deep-links into the ludamus sphere for
  enrollment (auth is Auth0 + per-sphere session cookies, so user-context
  actions have to happen on the ludamus domain anyway).

## WordPress as CMS

Recommended data layer: **WPGraphQL + WPGraphQL for ACF**, cribbing from
Vercel's official [`cms-wordpress` example](https://github.com/vercel/next.js/tree/canary/examples/cms-wordpress).
WPGraphQL is now a canonical WordPress.org plugin maintained under Automattic
(v2.x, actively released through 2026), so the "is this plugin going to die"
risk is low. One query per page for post + ACF + media, and codegen gives us
TypeScript types. Plain REST is the fallback if we want zero WP plugins, but
the content model below wants ACF anyway.

Content model (WP side):

| Content            | WP construct                              |
| ------------------ | ----------------------------------------- |
| News/announcements | posts                                     |
| Static pages       | pages (about, venue, accreditation, etc.) |
| Guests             | CPT + ACF (photo, bio, links)             |
| Sponsors/partners  | CPT + ACF (logo, tier, URL)               |
| Exhibitors         | CPT + ACF                                 |

WP-side plugin list: WPGraphQL, WPGraphQL for ACF, ACF, an on-demand
revalidation webhook (either the [On-Demand Revalidation plugin](https://wordpress.org/plugins/on-demand-revalidation/)
or a ~20-line `transition_post_status` mu-plugin that PUTs to
`/api/revalidate`), optionally wp-graphql-yoast-seo if we keep Yoast.
A "Headless Mode"-style plugin or `template_redirect` snippet 301s the old
frontend to the new domain while keeping wp-admin and `/graphql` alive.

Rendering/caching:

- All WP fetches tagged `wordpress`; publish/edit in WP → webhook →
  `revalidateTag('wordpress')`. Content cached indefinitely otherwise.
- Draft preview via Next.js Draft Mode + WP application passwords (the Vercel
  example implements this).
- Images: keep the WP media library, allow `wp.bachanaliafantastyczne.pl/wp-content/uploads/**`
  in `images.remotePatterns`, let Vercel image optimization do the rest.
- WP permalinks must be switched to "Post name" — the current site uses the
  legacy `/index.php/slug/` PATHINFO format, which breaks URI resolution in
  headless setups. Next.js should 301 the old `/index.php/...` URLs to the
  new routes to keep inbound links and SEO.

Gotchas already scouted (details in research notes): ACF fields are opt-in
per field group for GraphQL/REST exposure; Yoast emits canonical/og URLs
pointing at the WP domain and needs rewriting; sitemap/robots should be
generated by Next.js; CORS is a non-issue as long as all WP fetching happens
in server components.

## Ludamus integration

Current reality (from reading the repo): ludamus is a server-rendered Django
6 monolith. **No public REST/GraphQL API, `X-Frame-Options: DENY`, no CORS
headers.** Iframes and client-side fetching are both dead on arrival. The
`/mcp/` endpoint is superuser-only, not a public integration surface.

Options considered:

1. **Add a public programme feed to ludamus (recommended).** A read-only
   JSON endpoint, e.g. `GET /chronology/event/<slug>/programme.json`,
   returning agenda items with session, track, space, time slot, presenter,
   and enrollment counts. The GLIMPSE architecture makes this cheap: a new
   thin web gate calling an existing/new mill service that returns pydantic
   DTOs — views never touch models anyway. Since we control ludamus, this is
   an upstream feature, not a workaround, and other conventions
   (Kapitularz etc.) get it for free. ICS export could ride along later.
2. Server-side scraping of the public event page from Next.js — works
   (server-to-server ignores CORS/XFO) but means parsing Django templates.
   Fragile; only worth it as a stopgap demo before (1) lands.
3. Iframe/embed — blocked by XFO, and would look bad anyway. No.
4. Direct DB access — couples the website to ludamus internals. No.

Website side: a `/program` route fetches the feed in a server component with
a short ISR window (e.g. `revalidate: 300` — programme data changes during
the con, unlike WP content). The grid gets its own visual treatment matching
the new site; each item links to the session page on the ludamus sphere for
details/enrollment.

Enrollment and call-for-proposals stay entirely in ludamus — it already has
proposal categories, waitlists with offer/claim, anonymous enrollment, and an
organizer panel. The current WP "ZGŁOSZENIA PROGRAMU" page becomes a landing
page linking into the ludamus proposal flow. This needs a **Bachanalia sphere**
set up (spheres resolve by hostname, e.g. `bachanalia.zagrajmy.net`).

## Site map (v1)

Carried over from the current site, trimmed:

- `/` — hero (date/venue/edition), latest news, sponsors strip
- `/aktualnosci` (+ `/aktualnosci/[slug]`) — news from WP posts
- `/o-konwencie`, `/czas-i-miejsce`, `/organizator`, `/regulamin` — WP pages
- `/akredytacja` — pricing, from a WP page or ACF options
- `/program` — grid from ludamus feed
- `/zglaszanie-programu`, `/wolontariat` — landing pages → ludamus / form
- `/goscie` — guests CPT
- `/wystawcy` — exhibitors CPT
- `/wspieraja-nas` — sponsors CPT

Forms that stay outside ludamus (e.g. staff/volunteer signup, contact): a
Next.js server action posting to email or a spreadsheet is simpler than
keeping CF7 alive — decide per form.

## Phases

1. **Scaffold** — new repo from the Vercel `cms-wordpress` example: App
   Router, TypeScript, Tailwind, GraphQL codegen, revalidate route. Deploy to
   Vercel on a preview domain immediately.
2. **WP prep** — install WPGraphQL (+ACF), define CPTs/field groups, switch
   permalinks, set up application password, wire the revalidation webhook.
   Can happen on the live WP without affecting the current theme.
3. **Content pages** — news, static pages, guests/sponsors/exhibitors,
   layout/design, old-URL redirects.
4. **Ludamus feed** — upstream PR adding the programme JSON gate + tests;
   Bachanalia sphere setup; `/program` grid on the website.
5. **Cutover** — point apex DNS at Vercel, move WP to `wp.` subdomain,
   enable frontend redirect on WP, verify old URLs 301 correctly.
6. **Post-launch** — draft preview mode, OG images, con-time niceties
   ("happening now" on the programme page).

Phases 3 and 4 are independent and can run in parallel.
